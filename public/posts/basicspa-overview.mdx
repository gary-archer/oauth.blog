# Basic SPA - Overview

In the <a href='web-architecture-goals.mdx'>Web Architecture Goals</a> post I summarised some desired qualities. This blog's first code deliverable is an end-to-end web and API code sample that you can run on a development computer. The web application is a Single Page Application and the end-to-end flow integrates an SPA, API and authorization server.

### Code Sample Goal

The initial SPA code sample demonstrates secure, modern, productive OAuth development setups. The OAuth integration separates concerns so that web developers focus on the customer experience, while API developers protect data. Future code samples will extend the initial SPA and API further.

### Business Focused Development

The following table summarises the business-like behaviours that the initial code sample demonstrates.

| Behaviour | Description |
| --------- | ----------- |
| Authorization Server | Developers can connect to a cloud authorization server from code running on their computer. |
| SPA Security | The SPA uses strong browser security and uses the authorization code flow for logins. It then uses access tokens to call APIs. |
| API Security | The API validates JWT access tokens, then works with claims to implement authorization. |
| Reliable | Both the SPA and API handle expiry and other failure conditions in a solid manner. |
| Real World URLs | The SPA makes cross domain calls to the API on the local computer, and you can view its OAuth messages. |

The code sample uses a production-like setup on a development workstation. Identical technical behaviours would execute in deployed environments. At a real organization this would prevent mistakes and improve the time to market.

### Security Libraries

The SPA and API use the following respected open source libraries:

| Component | Library | Features |
| --------- | ------- | -------- |
| SPA | [oidc-client-ts](https://github.com/authts/oidc-client-ts) | Implements OAuth and OpenID Connect for browser-based clients, to enable user logins, token retrieval and session management. |
| API | [jose](https://github.com/panva/jose) | Validates JWT access tokens, returns claims and manages looking up and caching of token signing public keys. |

### Programming Languages

This blog uses multiple technologies, and they are all modern and cross platform. I start with these JavaScript based stacks, and use TypeScript for its extra language features and type checking benefits:

| Component | Language | Platforms |
| --------- | -------- | --------- |
| SPA | TypeScript | Chrome / Edge / Firefox / Safari / Mobile |
| API | Node.js + TypeScript | Windows / macOS / Linux |

For this blog's early SPAs I use vanilla TypeScript code, to avoid the overhead of more complex web frameworks. OAuth integration in popular web frameworks can work in the same way, as I illustrate in this blog's <a href='final-spa-overview.mdx'>Final SPA</a>, which uses the same patterns in React.

### Web and API Hosting

In OAuth, you should separate web and API hosting, so that API developers deal with data security and web developers focus on the frontend. This separation is a key factor in reducing development responsibilities to help enable a fast time to market. In deployed systems I represent this separation with an AWS custom domain of *authsamples.com* and the following base URLs:

| Component | Base URL |
| --------- | -------- |
| API | https://api.authsamples.com |
| Web Static Content | https://www.authsamples.com |

On a development computer I almost never use URLs like *http://localhost:3000* for application development. Instead, I use real-world domain names and HTTPS for web and API hosting. I use a *-dev* suffix for URLs to represent the local environment.

| Component | Base URL |
| --------- | -------- |
| API | https://api.authsamples-dev.com |
| Web Static Content | https://www.authsamples-dev.com |

The web static content host also applies a restrictive Content Security Policy (CSP), to protect against XSS threats. Developing with domain names encourages web developers to more fully understand browser behaviours and improve the security of their SPAs.

The use of domain names requires some initial infrastructure work. Once complete, you run with the same security as production systems, so you can avoid most deployment mistakes. You also have rich capabilities for testing OAuth flows on a development workstation.

### Code Sample Theme

This blog's code samples represent a fictional financial scenario, that pretends to display sensitive investment details. In a real financial app you would need to protect this type of data carefully:

![List View](../images/20/list-view.jpg?v=20251229)

The actual data is just made up and does not represent anything real. The SPA receives fixed JSON data from the API and renders it. The SPA views include a basic *Transactions View* and you can imagine that this contains money transaction details.

![Details View](../images/20/details-view.jpg?v=20251229)

This view primarily exists to demonstrate navigation and deep linking, where an access token must be retrieved before the SPA can present its view. Users can bookmark URLs to secured resources, and the SPA may need to renew a token or re-authenticate the user before presenting the view:
