# ID Tokens

Previously I explained some <a href='logout.mdx'>Logout</a> behaviours. I finish the discussion on SPA session management by describing the role of ID tokens, which are designed to be read by clients rather than APIs.

### ID Token Role

An ID token is a signed assertion from the authorization server, presenting proof of the authentication event. It can provide details like the authentication method(s), the authentication time and authentication-related attributes like the user's account status. Frontend applications can adjust their display based on these attributes.

### Receiving ID Tokens

An earlier post explained some <a href='basicspa-oauthworkflow.mdx'>Initial OAuth Messages</a>. Clients receive an authorization code in a front channel browser response, then send the code in a back channel HTTPS POST to the token endpoint of the authorization server, to swap the code for tokens. When you run an authorization code flow with *response_type=code* you receive all tokens, including the ID token, on the back channel.

### ID Token v Access Token v Refresh Token

Some basic website solutions use OAuth and OpenID Connect only partially, to authenticate the user. They then validate the ID token and start a cookie-based application session. The website interacts directly with a database to get data. The website may use ID token claims, like a user identity, to authorize access to data.

If you use OAuth and OpenID Connect correctly, the main application session is with APIs. The refresh token represents the lifetime of the API session. The main credential with which to make backend connections is the access token, which contains scopes and claims representing a business level of access.

|   | ID Token | Access Token | Refresh Token |
| - | -------- | ------------ | ------------- |
| Used At | Client | API | Authorization Server |
| Used For | Proof of Authentication | API Message Credential | Access Token Renewal |
| Lifetime | Expiry Not Used | Short Expiry | Long Expiry |
| Silently Renewable | No | Yes | Yes |

In OAuth, the ID token is not meant to be used for authorization. It should never leave the client, except in special use cases. If a client ever sends an ID token to an API, the API's scope and audience validation checks should immediately reject the token.

### ID Tokens and Privacy

OpenID Connect mandates that ID tokens have the JWT format, since they must be readable by frontend code. It is possible to include personal details such as *name* and *email* in ID tokens. I consider it safer and more correct to avoid personal data in ID tokens, which then sits around for the entire authenticated user session, so that there is more risk of leaking sensitive data.

For example, ID tokens are sometimes sent on the front-channel in logout redirects. Personal data might then end up in the browser history or server logs. Therefore, this blog's code samples instead download personal data from the OpenID Connect userinfo endpoint.

### Using ID Tokens

ID tokens contain proof of the authentication event and can also contain other attributes of use to frontend clients, like boolean flags about the user's account status. They are of far less significance than access tokens. This blog's frontend code samples allow security libraries to validate an ID token but otherwise ignore the ID token.

### Where Are We?

I summarised a few points about ID tokens, which provide verifiable proof of the authentication event to clients. All modern secured frontends should receive an ID token as part of using up-to-date security.
